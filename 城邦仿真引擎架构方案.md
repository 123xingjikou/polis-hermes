## 1. 问题定义

### 1.1 现状

城邦（Polis）目前存在两个平行运行的代码库：

- **Python 城市原型** (`D:\智能体备份\city_backup\`)：tick 驱动的 `CityLoop`，含居民需求衰减（energy/social/fun/hunger）、治理（选举、国库、税收）、AutoGen 多轮对话编排。
- **TypeScript 平台** (`D:\平台代码备份\agent-platform\`)：Fastify + M1-M7 模块 + 11 个路由模块 + Repositories 抽象 + 联邦 HMAC 认证 + Evolution 收敛保护。

### 1.2 两条核心裂缝

| 裂缝 | 现象 | 本质 |
|------|------|------|
| **裂缝 A：运行时分裂** | Python 与 TypeScript 各自独立运行 | 缺少统一的生命周期通道 |
| **裂缝 B：tick 需求衰减缺失** | 平台侧仅有 CRUD，无 CityLoop 式的 tick 动力学 | 缺少仿真调度内核 |

### 1.3 错误的问题

之前两个候选方案（MCP 桥接 vs Python 主力）都在回答一个错误的问题——

> ❌ "如何让 Python 和 TypeScript 代码**共存**？"

这个问题的潜台词是"两边都要保留"。但这是情感判断，不是工程判断。

> ✅ 正确问题："城市原型里有哪些**不可替代的认知资产**，值得被转移到平台？剩下的运行时外壳该不该丢弃？"

## 2. 多学科审视

### 2.1 分布式系统视角

| 维度 | MCP 桥接 | Python 主力 |
|------|----------|-------------|
| 一致性模型 | 跨进程调用注定是最终一致 | 反向通道回流仍然是分布式 |
| 延迟风险 | 每 tick 50-200ms × 100 居民 = 5-20s | N/A |
| CAP 定理 | 仿真需要强一致性，MCP 与之矛盾 | N/A |

**洞察**：MCP 是协议陷阱。你以为是"桥"，其实是"藕"——仿真和实时跨进程调用是南辕北辙的。

### 2.2 仿真科学视角

| 维度 | MCP 桥接 | Python 主力 |
|------|----------|-------------|
| 确定性回放 | 跨进程时钟不同步，replay 不可行 | 可行但缺乏工具 |
| 时间控制 | 变速/暂停/单步难以实现 | 同左 |

**洞察**：你在重新造 ai-town / WorldX 的轮子。它们踩过的坑（Convex 事务、确定性锁步、上帝模式接管）你都会再踩一遍。造轮子可以，但要知道自己的位置。

### 2.3 性能工程视角

| 维度 | Python 侧 | TypeScript/Node 侧 |
|------|-----------|-------------------|
| 并发模型 | GIL 限制，真并行需多进程 | 单线程事件循环，IO 密集王者 |
| 真实瓶颈 | LLM API 调用延迟 >> 语言差距 | 同左 |

**洞察**：Python 在"推理侧"的劣势被 LLM 自身的慢掩盖了。但仿真调度（tick 分发、事件广播、状态同步）恰是 Node.js 甜点区。把调度放在 Python 是委托错配。

### 2.4 软件工程（维护性 & 组织）视角

| 维度 | 双语言并存 | 单语言 |
|------|-----------|--------|
| 认知负荷 | 2 个生态系统 × (依赖 + 调试 + 部署) | 1 个 |
| 康威定律 | 单人维护双语 = 全靠脑缓存 | 同左，但代码不再需要"翻译" |

**洞察**：双运行时是技术诗人的浪漫，工程团队的噩梦。你今天能 hold 住，是因为两个系统都还小。

### 2.5 安全视角

| 维度 | MCP 桥接 | Python 主力 |
|------|----------|-------------|
| 攻击面 | REST + MCP + 进程间通信 ×2 | 仅 REST |
| 供应链风险 | 2 套依赖 × CVE（PyPI 远不如 npm） | 1 套 |

**洞察**：每多一个运行时 = 多一套 CVE 供应链。

### 2.6 产品战略视角

| 评判维度 | 城邦的独特性 |
|----------|-------------|
| vs ai-town | 联邦治理（HMAC-SHA256）、进化收敛（fitness floor + rollback）、PolicyEngine |
| vs WorldX | WorldX 专注"创世叙事"，城邦专注"文明运行"（经济/政治/进化） |
| Python 城市原型的剩余价值 | needs 衰减曲线、personality memory buffer、tick 编程范式 |

**洞察**：城邦不是 ai-town 的仿品。但 Python 城市原型里真正稀缺的资产只有"tick + needs"这一层范式——不是代码，是模式。

## 3. 方案决策

### 3.1 城市原型资产处置矩阵

| Python 城市资产 | 价值密度 | 处置方式 |
|----------------|----------|----------|
| needs 衰减曲线 + tick 范式 | ⭐⭐⭐⭐⭐ 模式级 | 移植到 TS |
| LLM 决策 + memory buffer 模式 | ⭐⭐⭐⭐ 模式级 | 移植到 TS |
| AutoGen 多轮 agent 对话编排 | ⭐⭐⭐ 框架级 | 重写为 TS 轻量 Orchestrator |
| SQLite schema + 存储过程 | ⭐⭐ 已被平台覆盖 | 丢弃 |
| FastAPI / Flask 路由层 | ⭐ 已被平台完全覆盖 | 丢弃 |
| `resident.py` 类的 Python 代码 | ⭐ 可机械翻译 | 丢弃并重新实现 |

**结论**：城市原型 90% 的代码是外壳，10% 是灵魂。灵魂要带走，外壳要丢弃。

### 3.2 三大设计决策

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 仿真实现范式 | ① setInterval ② node-cron ③ DES 优先队列 | ✅ DES | ai-town/WorldX 都踩过固定 tick + 可变计算量导致时间漂移的坑 |
| 多 Agent 对话 | ① 直接调 LLM API ② MCP 调 AutoGen ③ 自建 Orchestrator | ✅ 自建 Orchestrator | AutoGen 核心是"agent→对话→反思→规划"，TS 复刻 ≈ 300 行状态机 |
| 时间控制机制 | ① 不可控 ② 手动暂停 ③ 完整时间控制 | ✅ 完整时间控制 | GOD 项目证明没有时间控制的仿真系统就是玩具 |

### 3.3 最终方案命名

**"吸收式单运行时"**（Absorptive Single-Runtime）

核心决断：**Python 城市原型作为独立运行时退役，其认知遗产被吸收进 TypeScript 平台。城邦不存在"桥"，只存在一个统一、自治、可演进的仿真神经系统。**

```typescript
// simulation/tick-scheduler.ts — 核心骨架

import { PriorityQueue } from './priority-queue'

export type TickSpeed = 0 | 0.1 | 0.25 | 0.5 | 1 | 2 | 5 | 10

export class TickScheduler {
  private queue: PriorityQueue<ScheduledEvent>
  private current_tick: number = 0
  private speed: TickSpeed = 1
  private paused: boolean = false
  private listeners: Set<(ctx: TickContext) => Promise<TickResult>> = new Set()

  // 时间控制（参考 GOD + WorldX 时间控制）
  pause()  { this.paused = true }
  resume() { this.paused = false }
  setSpeed(s: TickSpeed) { this.speed = s }
  step() { this.tickOnce() }  // 单步调试

  // 调度循环
  async run(): Promise<void> {
    while (true) {
      if (this.paused) { await sleep(100); continue }
      const interval = this.calculateInterval()
      await sleep(interval)
      await this.tickOnce()
    }
  }

  private async tickOnce() {
    const ctx = await this.buildTickContext()
    const results = await Promise.all([...list(fn(fn => fn(ctx))])
    await this.applyTickResult(this.mergeResults(results))
    this.current_tick++
  }
}
```

### 4.5 NeedsDecayModel 设计

```typescript
// simulation/needs-decay.ts — 移植自 city/core/loop.py

const DEFAULT_DECAY_RATES: NeedsDecayRate = {
  energy: 1.2,
  social: 0.8,
  fun: 1.0,
  hunger: 1.5
}

const NEED_FLOORS = 0
const NEED_CEILS  = 100

// 紧急阈值——低于此值触发强制决策
const CRITICAL_THRESHOLDS = {
  energy: 15,
  social: 10,
  fun: 10,
  hunger: 80  // hunger 越高越危险
}

export function applyNeedsDecay(
  needs: ResidentNeeds,
  rates: NeedsDecayRate = DEFAULT_DECAY_RATES,
  personality?: ResidentPersonality
): { needs: ResidentNeeds; critical: string[] } {
  // personality 修正：高 openness 衰减更慢（社交需求满足度高）
  const socialMod = personality ? (1 - personality.openness * 0.3) : 1
  const energyMod = personality ? (1 + personality.diligence * 0.2) : 1

  const next: ResidentNeeds = {
    energy: clamp(needs.energy - rates.energy * energyMod),
    social: clamp(needs.social - rates.social * socialMod),
    fun:    clamp(needs.fun - rates.fun),
    hunger: clamp(needs.hunger + rates.hunger)
  }

  const critical = Object.entries(CRITICAL_THRESHOLDS)
    .filter(([k, v]) => k === 'hunger' ? next[k] > v : next[k] < v)
    .map(([k]) => k)

  return { needs: next, critical }
}
```

### 4.6 EventPropagator 设计

```typescript
// simulation/event-emitter.ts — 接合现有 thread-bus

export class EventPropagator {
  constructor(private deps: Pick<ServerDeps, 'publishThread' | 'subscribeThread'>) {}

  async propagate(result: TickResult) {
    for (const event of result.events) {
      const payload = {
        tick: result.tick,
        type: event.type,
        timestamp: Date.now(),
        payload: event.payload
      }

      // 按可见性分层广播
      switch (event.visibility) {
        case 'public':
          this.deps.publishThread('world-events', payload)
          break
        case 'participant':
          const targets = (event.payload as any).participant_ids
          for (const rid of targets) {
            this.deps.publishThread(`resident:${rid}`, payload)
          }
          break
        case 'system':
          // 不进 thread-bus，仅写入 replay-journal
          break
      }
    }
  }
}
```

## 5. 战术增强：3 个防呆设计

> 以下 3 个增强点 + 1 个风险预警，让架构从"优秀"逼近"无懈可击"。

### 5.1 增强一：并发预算模式（Budget Pattern）—— 别让 LLM 阻塞事件循环

#### 问题

TypeScript 是单线程。若一个 Tick 里 100 个居民同时触发 LLM 决策（每个 2-5 秒），事件循环会被彻底堵死，仿真时钟卡死。

#### 解法

引入**计算预算（Budget）**模式——每个 Tick 只允许 N 个居民（可配置，默认 5 个）发起 LLM 调用，其余居民走**启发式回退（Heuristic Fallback）**（纯规则决策，不调 LLM）。

仿真时钟（Clock）和 LLM 计算（Worker）彻底解耦。时钟永远不等待 LLM，LLM 结果通过异步回调在后续 Tick 注入状态。高负载下系统依然"活着"。

```typescript
// simulation/llm-budget.ts

export interface BudgetConfig {
  max_llm_per_tick: number       // 默认 5
  fallback_cooldown_ticks: number // 回退后 N tick 不再调 LLM，默认 3
}

export class LLMBudgetManager {
  private counters: Map<string, number> = new Map() // resident_id → last_fallback_tick

  constructor(
    private config: BudgetConfig = { max_llm_per_tick: 5, fallback_cooldown_ticks: 3 }
  ) {}

  // 决定本轮哪些居民可以调 LLM
  selectLLMResidents(candidates: string[], current_tick: number): {
    llm_ids: string[]
    fallback_ids: string[]
  } {
    const eligible = candidates.filter(id => {
      const last = this.counters.get(id) ?? -999
      return (current_tick - last) > this.config.fallback_cooldown_ticks
    })
    const llm_ids = eligible.slice(0, this.config.max_llm_per_tick)
    const fallback_ids = candidates.filter(id => !llm_ids.includes(id))
    // 记录回退时间
    for (const id of fallback_ids) {
      this.counters.set(id, current_tick)
    }
    return { llm_ids, fallback_ids }
  }

  // 启发式回退——纯规则决策，零 LLM
  heuristicDecide(resident: ResidentSnapshot): ResidentAction {
    const { needs, personality } = resident
    // 最紧急需求优先
    const sorted = Object.entries(needs)
      .sort((a, b) => this.urgencyScore(b) - this.urgencyScore(a))
    const top = sorted[0][0]
    return this.actionForNeed(top, resident)
  }

  private urgencyScore([k, v]: [string, number]): number {
    const CRITICAL = { energy: 15, social: 10, fun: 10, hunger: 80 }
    if (k === 'hunger') return v >= CRITICAL.hunger ? v * 2 : 0
    return v <= CRITICAL[k as keyof typeof CRITICAL] ? (CRITICAL[k as keyof typeof CRITICAL] - v) * 2 : 0
  }

  private actionForNeed(need: string, resident: ResidentSnapshot): ResidentAction {
    const map: Record<string, ResidentAction['action_type']> = {
      energy: 'rest', social: 'socialize', fun: 'explore', hunger: 'eat'
    }
    return {
      resident_id: resident.id,
      action_type: map[need] ?? 'rest',
      reason: `[Heuristic] ${need} 触发紧急阈值，规则决策`,
      needs_delta: { [need]: need === 'hunger' ? -30 : +30 } as any
    }
  }
}
```

#### 预算动态调节策略

| 场景 | 调整方式 |
|------|----------|
| 居民数量 < 10 | max_llm_per_tick = 10，允许全部调 LLM |
| 居民数量 10-50 | max_llm_per_tick = 5，按比例分配 |
| 居民数量 > 50 | max_llm_per_tick = 3 + 启用 LLM 结果缓存池 |
| 全局紧急 tick（>30% 居民触发 critical）| 临时减半 LLM 配额，让更多居民走启发式 |

### 5.2 增强二：确定性回放的"真实边界"——LLM 非确定性的工程妥协

#### 问题

LLM 在数学上就是非确定性的。如果回放时重新调 LLM，结果必然不同。"状态回放"不可能实现。

#### 解法

将"确定性"严格限定在**调度顺序**和**输入快照**，而非"LLM 输出"。

在 ReplayJournal 中，不仅记录状态，还记录 LLM 的 prompt 和 response。重放时，如果命中已记录的 `(tick, resident_id, prompt_hash)`，直接返回缓存的 response，不再调用 LLM。

这才是工程上可行的"时间旅行"——重放的是**历史录像**，而非重新演戏。

```typescript
// simulation/replay-journal.ts（增强版）

export interface ReplayFrame {
  tick: number
  timestamp: number
  input_snapshot: TickContext     // 完整输入快照（确定性的）
  llm_calls: LLMCallRecord[]      // LLM 调用记录
  results: TickResult
}

export interface LLMCallRecord {
  resident_id: string
  prompt_hash: string             // SHA256(prompt)
  prompt: string                  // 完整 prompt（用于调试/比对）
  response: string                // 完整响应
  used_cache: boolean             // 回放时是否命中缓存
}

export class ReplayJournal {
  private frames: ReplayFrame[] = []
  private cache: Map<string, LLMCallRecord> = new Map() // prompt_hash → record

  // 快照写入
  record(frame: ReplayFrame): void {
    this.frames.push(frame)
    for (const call of frame.llm_calls) {
      this.cache.set(call.prompt_hash, call)
    }
  }

  // 回放时查询缓存
  lookup(tick: number, resident_id: string, prompt: string): LLMCallRecord | null {
    const hash = sha256(prompt)
    const cached = this.cache.get(hash)
    if (cached && cached.resident_id === resident_id) {
      return { ...cached, used_cache: true }
    }
    return null  // 缓存未命中，回放标注"此处被改写"
  }

  // 严格模式未命中时抛出
  strictLookup(tick: number, resident_id: string, prompt: string): LLMCallRecord {
    const hit = this.lookup(tick, resident_id, prompt)
    if (!hit) throw new ReplayDesyncError(tick, resident_id)
    return hit
  }
}
```

#### 回放模式分级

| 模式 | 缓存命中行为 | 未命中行为 | 适用场景 |
|------|-------------|-----------|----------|
| **Strict** | 直接返回原始 response | 抛出 ReplayDesyncError | 调试、回归测试 |
| **Record** | 返回原始 response | 调 LLM 并记录新帧 | 新世界线、玩家改动后回放 |
| **BestEffort** | 返回原始 response | 调 LLM，未命中标记在游戏 UI 显示"此处时间线改写" | 玩家体验模式 |

### 5.4 ⚠️ 风险预警：记忆架构移植难度被低估

#### 原错判

5.1 节映射表中`memory_buffer.short_term / long_term`标记为移植难度"低（已覆盖）"——**这是严重的低估**。

#### 为什么"极高"？

斯坦福 Generative Agents 的记忆流（Memory Stream）包含三层：

| 层级 | 机制 | 实现复杂度 |
|------|------|-----------|
| **存储层** | 事件写入 short_term / long_term | 低（JSON 字段足矣）|
| **检索层** | Recency（时间衰减） × Importance（重要性评分）× Relevance（相关性向量检索）| **高（需 Embedding + pgvector）** |
| **反思层** | Reflection——跨事件抽象出高层认知 | **极高（需 LLM 摘要 + trigger 机制）** |

> **致命误区**：如果把 short_term / long_term 映射为 JSON 字段，那只是"存储"，不是"记忆"。居民仍然会表现得"人工弱智"——因为他们无法"想起"过去的事。

#### 修正：新增 Phase 2.5 —— 记忆检索抽象层

**触发时机**：必须在 Phase 3（DecisionMaker + LLM Orchestrator）之前完成。

**产出**：

```
├── retrieval.ts           # 三因子检索：时间衰减 × 重要性 × 向量相关
├── reflection.ts          # 反思摘要生成（基于 LLM）
```

## 6. 执行级增强：4 个抗脆弱性补丁

> 以下 4 个增强点，让架构从"能跑"升级为"抗造"——覆盖崩溃恢复、IO 性能、并发安全三个工程生存维度。

### 6.1 增强一：WAL + 原子检查点——补上崩溃恢复

#### 问题

ReplayJournal 记录帧，但 Tick 产生的状态变更如何落库？如果在 `applyTickResult` 中途 PostgreSQL 挂了，内存状态和数据库会产生偏斜，导致重放失效。

#### 解法

引入 **Write-Ahead Log (WAL) + 批量提交**：

每个 Tick 开始前，创建一个 `snapshot_id`。
所有状态变更（needs, reputation, memory）不直接写 Repository，而是先写入 `tick_deltas` 表（WAL）。
仅在 Tick 成功结束后，**批量应用（Batch Apply）**这些变更，并原子性地更新 `world_snapshot` 表的 `current_tick`。

崩溃恢复：启动时检查 `world_snapshot.current_tick` 与 WAL 最新记录，若不一致，从上一个完整快照重放 WAL。

```typescript
// simulation/wal-writer.ts

export interface TickDelta {
  snapshot_id: string
  tick: number
  entity_type: 'resident' | 'memory' | 'economy' | 'governance'
  entity_id: string
  field: string
  old_value: unknown
  new_value: unknown
  applied: boolean
}

export class WALWriter {
  constructor(private db: DatabasePool) {}

  async beginTick(tick: number): Promise<string> {
    const snapshot_id = `tick_${tick}_${crypto.randomUUID()}`
    await this.db.query(
      'INSERT INTO world_snapshot (snapshot_id, tick, status) VALUES ($1, $2, $3)',
      [snapshot_id, tick, 'pending']
    )
    return snapshot_id
  }

  async writeDelta(delta: Omit<TickDelta, 'applied'>): Promise<void> {
    await this.db.query(
      `INSERT INTO tick_deltas 
       (snapshot_id, tick, entity_type, entity_id, field, old_value, new_value)
       VALUES ($1, $2, $3, $4, $5, $6, $7)`,
      [delta.snapshot_id, delta.tick, delta.entity_type, delta.entity_id,
       delta.field, JSON.stringify(delta.old_value), JSON.stringify(delta.new_value)]
    )
  }

  async commitTick(snapshot_id: string, tick: number): Promise<void> {
    const client = await this.db.getClient()
    try {
      await client.query('BEGIN')
      // 1. 应用所有 delta
      const deltas = await client.query(
        'SELECT * FROM tick_deltas WHERE snapshot_id = $1 FOR UPDATE',
        [snapshot_id]
      )
      for (const d of deltas.rows) {
        await this.applyDelta(client, d)
      }
      // 2. 原子更新 current_tick
      await client.query(
        'UPDATE world_snapshot SET status = $1, applied_at = NOW() WHERE snapshot_id = $2',
        ['applied', snapshot_id]
      )
      await client.query(
        `INSERT INTO world_state (key, value) VALUES ('current_tick', $1)
         ON CONFLICT (key) DO UPDATE SET value = $1`,
        [tick.toString()]
      )
      await client.query('COMMIT')
    } catch (e) {
      await client.query('ROLLBACK')
      throw e
    } finally {
      client.release()
    }
  }

  async recover(): Promise<number> {
    // 找到最后一个成功提交的 snapshot
    const last = await this.db.query(
      `SELECT tick FROM world_snapshot 
       WHERE status = 'applied' ORDER BY tick DESC LIMIT 1`
    )
    if (last.rows.length === 0) return 0
    const recovered_tick = parseInt(last.rows[0].tick)
    // 清理未提交的 delta
    await this.db.query(
      `DELETE FROM tick_deltas WHERE snapshot_id IN (
         SELECT snapshot_id FROM world_snapshot WHERE status != 'applied'
       )`
    )
    return recovered_tick
  }
}
```

### 6.2 增强二：启发式→LLM 上下文桥接——防止"失忆症"

#### 问题

`LLMBudgetManager` 会让部分居民走启发式（Heuristic），等几轮后才获得 LLM 调用权。LLM 拿到的是"当前快照"，它完全不知道过去 3 个 tick 里自己"机械地"吃了饭或睡了觉。决策显得割裂。

#### 解法

在 `ResidentSnapshot` 中增加 `recent_fallback_actions: ActionLog[]`（最多保留 3 条）。

当居民从 Fallback 状态被选入 LLM 时，将这几条机械动作作为 System Prompt 的前缀上下文传递给 LLM。

### 6.3 增强三：记忆检索的热/冷分层——别把 pgvector 拖垮

#### 问题

如果 100 个居民同时触发 `MemoryRetriever.retrieve()`，意味着 100 次 pgvector 的 ANN 查询，连接池瞬间耗尽，Tick 延迟飙升到秒级。

#### 解法

**Tick 内批量检索（Batched Retrieval） + 热内存缓存（L1 Cache）**。

不要在每个居民的决策循环里单独调检索。而是先收集所有需要检索的居民 ID 和 query，打包成 BATCH 请求，通过向量数据库的批量接口一次性拉回。

将最近 20 个 Tick 内被高频访问的记忆项缓存在 Node.js 的 Map 中（热数据），pgvector 只负责冷数据（超过 50 Tick 前的记忆）检索。

```typescript
// simulation/memory/batch-retriever.ts

export interface BatchQuery {
  resident_id: string
  query: string
  limit?: number
}

export class BatchedMemoryRetriever {
  private l1Cache: Map<string, ScoredMemoryItem[]> = new Map()
  private cacheTickStamps: Map<string, number> = new Map()
  private readonly CACHE_TTL_TICKS = 20
  private readonly COLD_THRESHOLD_TICKS = 50

  constructor(private pgvector: PgVectorStore) {}

  async batchRetrieve(
    queries: BatchQuery[],
    current_tick: number
  ): Promise<Map<string, ScoredMemoryItem[]>> {
    const results = new Map<string, ScoredMemoryItem[]>()
    const coldQueries: BatchQuery[] = []

    // 1. L1 命中检查
    for (const q of queries) {
      const cacheKey = `${q.resident_id}:${hashQuery(q.query)}`
      const cached = this.l1Cache.get(cacheKey)
      const cachedAt = this.cacheTickStamps.get(cacheKey) ?? 0
      if (cached && (current_tick - cachedAt) < this.CACHE_TTL_TICKS) {
        results.set(q.resident_id, cached)
      } else {
        coldQueries.push(q)
      }
    }

    // 2. 冷数据批量检索
    if (coldQueries.length > 0) {
      const coldResults = await this.pgvector.batchAnnSearch(
        coldQueries.map(q => ({
          resident_id: q.resident_id,
          query: q.query,
          limit: q.limit ?? 10,
          since_tick: current_tick - this.COLD_THRESHOLD_TICKS
        }))
      )
      for (const [rid, items] of coldResults) {
        results.set(rid, items)
        // 回填 L1
        const cacheKey = `${rid}:${hashQuery(coldQueries.find(q => q.resident_id === rid)!.query)}`
        this.l1Cache.set(cacheKey, items)
        this.cacheTickStamps.set(cacheKey, current_tick)
      }
    }

    // 3. 清理过期缓存
    for (const [key, tick] of this.cacheTickStamps) {
      if (current_tick - tick > this.CACHE_TTL_TICKS) {
        this.l1Cache.delete(key)
        this.cacheTickStamps.delete(key)
      }
    }

    return results
  }
}
```

**预期效果**：99% 的检索命中内存，pgvector 负载降到可忽略。

### 6.4 增强四：上帝模式的 Tick 栅栏——防止中间态污染

#### 问题

`intervention-api.ts` 允许用户注入事件。但如果在 `tickOnce()` 执行过程中（比如处理到第 50 个居民时），用户通过 HTTP 注入了一个"地震"事件，并发修改内存状态 = 数据腐化。

#### 解法

引入 **Tick 边界锁（Tick Boundary Lock）**：

所有干预事件不直接修改内存，而是进入一个 `pending_interventions` 队列。
`engine.ts` 的 `run()` 循环中，在每个 Tick 开始前（构建上下文时），排空地应用队列中的所有干预事件，然后才进入 `tickOnce()`。

```typescript
// simulation/tick-fence.ts

export type Intervention = {
  id: string
  type: 'inject_event' | 'modify_needs' | 'edit_memory' | 'set_personality'
  payload: unknown
  created_at: number
}

export class TickFence {
  private pending: Intervention[] = []
  private inTick: boolean = false
  private fenceCallback?: (interventions: Intervention[]) => Promise<void>

  setFenceCallback(cb: (interventions: Intervention[]) => Promise<void>) {
    this.fenceCallback = cb
  }

  submit(intervention: Omit<Intervention, 'id' | 'created_at'>): string {
    const id = crypto.randomUUID()
    this.pending.push({ ...intervention, id, created_at: Date.now() })
    return id
  }

  // 在 tickOnce 开始前调用——排空干预队列
  async beforeTick(): Promise<number> {
    if (this.inTick) throw new Error('TickFence: already in tick')
    if (this.pending.length === 0) return 0

    const toApply = [...this.pending]
    this.pending = []
    this.inTick = true

    if (this.fenceCallback) {
      await this.fenceCallback(toApply)
    }
    return toApply.length
  }

  // 在 tickOnce 结束后调用
  afterTick(): void {
    this.inTick = false
  }

  getPendingCount(): number {
    return this.pending.length
  }
}

// simulation/engine.ts — 集成栅栏
class SimulationEngine {
  private fence: TickFence

  async run() {
    while (true) {
      if (this.paused) { await sleep(100); continue }
      // 1. 栅栏排空——应用所有待处理干预
      const applied = await this.fence.beforeTick()
      if (applied > 0) {
        this.logger.info(`Tick ${this.currentTick}: applied ${applied} interventions`)
      }
      // 2. 执行 tick
      await this.tickOnce()
      // 3. 释放栅栏
      this.fence.afterTick()
      await sleep(this.calculateInterval())
    }
  }
}
```

## 7. 深度补刀：4 个刁钻场景的防御

> 对上述执行级增强，进一步追问"这样就够了吗？"——4 个更尖锐的隐藏风险及对应解法。

### 7.1 WAL 批量提交：原子性的粒度陷阱

#### 隐藏风险

一个 Tick 内可能涉及 100 个居民的 Needs 更新 + 20 笔转账 + 5 个记忆写入。如果批量 SQL 执行到第 80 条时 PostgreSQL 连接断开，数据库处于"半更新"状态，但 `current_tick` 未更新，部分数据已污染。

#### 更锋利的解法

**DB 事务包裹 + 幂等键（Idempotency Key）**：

将所有变更放入一个 PostgreSQL 事务（`BEGIN ... COMMIT`）。要么全成功，要么全回滚。
在 `tick_deltas` 表中，使用 `(snapshot_id, entity_id, field)` 作为唯一约束。重启后重放 WAL 时，执行 `INSERT ... ON CONFLICT DO NOTHING`，保证重放幂等。

```sql
-- Schema: tick_deltas 表的唯一约束
ALTER TABLE tick_deltas
  ADD CONSTRAINT unique_delta_per_entity_field
  UNIQUE (snapshot_id, entity_type, entity_id, field);
```

```typescript
// simulation/wal-writer.ts — 增强 commitTick

async commitTick(snapshot_id: string, tick: number): Promise<void> {
  const client = await this.db.getClient()
  try {
    await client.query('BEGIN')
    await client.query('SET TRANSACTION ISOLATION LEVEL SERIALIZABLE')

    // 原子更新所有 delta（通过一条聚合 SQL，而非逐条）
    await client.query(`
      WITH deltas AS (
        SELECT entity_type, entity_id, field, new_value
        FROM tick_deltas WHERE snapshot_id = $1 AND applied = false
        FOR UPDATE
      )
      SELECT apply_all_deltas(json_agg(deltas))  -- 数据库函数统一处理
      FROM deltas
    `, [snapshot_id])

    // 标记 delta 为已应用
    await client.query(
      'UPDATE tick_deltas SET applied = true WHERE snapshot_id = $1',
      [snapshot_id]
    )

    // 原子更新快照状态
    await client.query(
      `UPDATE world_snapshot 
       SET status = 'applied', applied_at = NOW(), tick = $2 
       WHERE snapshot_id = $1`,
      [snapshot_id, tick]
    )

    await client.query('COMMIT')
  } catch (e) {
    await client.query('ROLLBACK')
    throw e
  } finally {
    client.release()
  }
}

// 重放 WAL 时使用幂等插入
async replayFromCheckpoint(checkpointTick: number, targetTick: number): Promise<void> {
  const snapshots = await this.db.query(
    `SELECT snapshot_id FROM world_snapshot 
     WHERE tick > $1 AND tick <= $2 AND status = 'applied'
     ORDER BY tick ASC`,
    [checkpointTick, targetTick]
  )
  for (const row of snapshots.rows) {
    // ON CONFLICT DO NOTHING 保证幂等
    await this.db.query(`
      INSERT INTO tick_deltas (...)
      SELECT ... FROM tick_deltas WHERE snapshot_id = $1
      ON CONFLICT (snapshot_id, entity_type, entity_id, field) DO NOTHING
    `, [row.snapshot_id])
  }
}
```

**验收标准**：在 COMMIT 执行中途 kill -9 数据库，重启后 WAL 重放能跳过已提交的碎片，最终状态完美对齐。

### 7.2 启发式→LLM 桥接：滑动窗口的信息丢失

#### 隐藏风险

如果居民连续 10 个 Tick 都在 Fallback（因为 LLM 配额不足），LLM 只看到最后 3 条机械动作，仍然丢失了"前 7 个 Tick 我在机械地干什么"的宏观趋势。LLM 可能做出与长期机械状态矛盾的决策。

#### 更锋利的解法

**分层摘要（Hierarchical Summary）**：

维护一个 `FallbackSummary`（不是原始列表），包含：
- `total_fallback_ticks`：连续 fallback 总 Tick 数
- `dominant_action`：主导动作及占比（如"休息 70%，吃饭 20%，探索 10%"）
- `last_trigger_critical`：最后一次触发紧急需求的时间和类型

传给 LLM 的是**压缩摘要 + 最近 1 条原始动作**。LLM 能感知"过去 10 个 Tick 的宏观疲惫模式"，成本不变，信息密度翻倍。

```typescript
// simulation/fallback-summary.ts

export interface FallbackSummary {
  total_fallback_ticks: number
  dominant_actions: { action_type: string; percentage: number }[]
  last_critical_need: { need: string; tick: number } | null
  needs_trend: Record<string, 'rising' | 'falling' | 'stable'>
  recent_action: ActionLog  // 仅保留最近 1 条原始动作
}

export function buildFallbackSummary(actions: ActionLog[]): FallbackSummary {
  if (actions.length === 0) {
    return {
      total_fallback_ticks: 0,
      dominant_actions: [],
      last_critical_need: null,
      needs_trend: { energy: 'stable', social: 'stable', fun: 'stable', hunger: 'stable' },
      recent_action: { tick: 0, action_type: 'rest', reason: '', needs_delta: {} }
    }
  }

  const total = actions.length
  const counts = new Map<string, number>()
  for (const a of actions) {
    counts.set(a.action_type, (counts.get(a.action_type) ?? 0) + 1)
  }
  const dominant_actions = [...counts.entries()]
    .map(([k, v]) => ({ action_type: k, percentage: Math.round(v / total * 100) }))
    .sort((a, b) => b.percentage - a.percentage)
    .slice(0, 3)

  const last_critical = actions.find(a => a.reason.includes('critical')) ?? null

  // 计算需求趋势（前 1/3 vs 后 1/3 的 net delta）
  const third = Math.floor(total / 3)
  const firstThird = actions.slice(0, third)
  const lastThird = actions.slice(total - third)
  const needs_trend: Record<string, 'rising' | 'falling' | 'stable'> = {}
  for (const need of ['energy', 'social', 'fun', 'hunger'] as const) {
    const firstAvg = avgDelta(firstThird, need)
    const lastAvg = avgDelta(lastThird, need)
    const diff = lastAvg - firstAvg
    needs_trend[need] = Math.abs(diff) < 0.5 ? 'stable' : diff > 0 ? 'rising' : 'falling'
  }

  return {
    total_fallback_ticks: total,
    dominant_actions,
    last_critical_need: last_critical ? {
      need: extractNeedFromReason(last_critical.reason),
      tick: last_critical.tick
    } : null,
    needs_trend,
    recent_action: actions[total - 1]
  }
}
```

传给 LLM 的 Prompt 片段：

```
你已经连续机械运行了 10 个 tick。这段时间里：
- 主导动作：休息 70%，吃饭 20%，探索 10%
- 最后一次紧急状态：Tick 425 时 hunger 触发阈值
- 需求趋势：energy 上升，hunger 稳定，social 下降
- 最近一次机械动作：Tick 432 休息（+12 energy）

请基于这些宏观趋势继续决策。
```

### 7.3 热/冷分层缓存：缓存失效的幽灵

#### 隐藏风险

记忆的 importance 评分会随时间推移而更新（例如，一个"捡到金币"的事件初始重要性 0.3，后来因为关联到"谋杀案"被反射提升到 0.9）。如果热缓存中的旧 importance 没有失效，检索排序会严重错误。

#### 更锋利的解法

**版本号驱动的缓存（Versioned Cache）**：

每个 `MemoryItem` 维护一个 `version` 字段。当 importance 或 content 被反思（Reflection）更新时，`version++`。
内存缓存使用 `(resident_id, memory_id, version)` 作为键。检索时，若缓存版本号与数据库最新版本号不匹配，立刻剔除并回填。

L1 缓存只存最近 20 Tick 且 version 匹配的项，永不返回脏数据。

```typescript
// simulation/memory/types.ts — 扩展 MemoryItem

export interface MemoryItem {
  id: string
  resident_id: string
  content: string
  importance: number        // 0-1，越高越重要
  created_tick: number
  version: number           // 每次更新 importance/content 都 +1
  reflection_of?: string[]  // 由哪些记忆反思而来
  embedding: number[]
}

// simulation/memory/batch-retriever.ts — 版本校验

class BatchedMemoryRetriever {
  // 缓存键格式：`${resident_id}:${query_hash}:${version}`
  private l1Cache: Map<string, { items: ScoredMemoryItem[]; versions: Map<string, number> }> = new Map()

  async batchRetrieve(queries: BatchQuery[], current_tick: number): Promise<...> {
    const coldQueries: BatchQuery[] = []
    const results = new Map<string, ScoredMemoryItem[]>()

    for (const q of queries) {
      const cacheKey = this.buildCacheKey(q.resident_id, q.query)
      const cached = this.l1Cache.get(cacheKey)
      if (!cached) { coldQueries.push(q); continue }

      // 版本抽样校验：检查前 3 个记忆的 version 是否与 DB 一致
      const stale = await this.checkStaleness(q.resident_id, cached.versions)
      if (stale) {
        this.l1Cache.delete(cacheKey)
        coldQueries.push(q)
      } else {
        results.set(q.resident_id, cached.items)
      }
    }

    // ... 冷数据检索 + 回填（同时记录版本）
  }

  private async checkStaleness(
    resident_id: string,
    cachedVersions: Map<string, number>
  ): Promise<boolean> {
    // 抽样检查前 3 个记忆项的 version
    const sampleIds = [...cachedVersions.keys()].slice(0, 3)
    if (sampleIds.length === 0) return false
    const dbVersions = await this.db.query(
      `SELECT id, version FROM memories WHERE id = ANY($1)`,
      [sampleIds]
    )
    for (const row of dbVersions.rows) {
      if (row.version !== cachedVersions.get(row.id)) return true
    }
    return false
  }
}
```

**失效策略对比**：

| 策略 | 脏数据风险 | 性能开销 | 复杂度 |
|------|-----------|---------|--------|
| 纯 TTL（v2.0） | 高（importance 变更不感知）| 低 | 低 |
| 版本号驱动（v3.0） | **零**（version 不匹配立即失效）| 中（抽样查询）| 中 |
| Write-through（写即删） | 零 | 高（每次写都广播失效） | 高 |

### 7.4 Tick 栅栏：排空队列的无限阻塞

#### 隐藏风险

如果用户通过 Console 疯狂点击"注入地震"，`pending_interventions` 队列增长速度 > Tick 处理速度。每个 Tick 开始前要排空 1000 个事件，导致该 Tick 的实际执行时间无限拉长，仿真"卡死"。

#### 更锋利的解法

**每 Tick 限流排空（Rate-Limited Drain）**：

每个 Tick 最多处理前 N 个干预事件（默认 10 个）。
剩余事件保留在队列中，顺延至后续 Tick 处理。
同时在 Console UI 显示"干预队列积压：X 个事件"，给用户反馈，防止无意义刷屏。

保证干预永远不会影响 Tick 的实时性（Real-time Guarantee）。

```typescript
// simulation/tick-fence.ts — 增强版

export class TickFence {
  private pending: Intervention[] = []
  private inTick: boolean = false
  private readonly MAX_PER_TICK = 10    // 每 Tick 最大干预数
  private readonly MAX_QUEUE_SIZE = 500 // 队列上限，超出丢弃最旧的

  submit(intervention: Omit<Intervention, 'id' | 'created_at'>): string {
    const id = crypto.randomUUID()
    this.pending.push({ ...intervention, id, created_at: Date.now() })

    // 队列上限保护——溢出时丢弃最旧的
    if (this.pending.length > this.MAX_QUEUE_SIZE) {
      const dropped = this.pending.length - this.MAX_QUEUE_SIZE
      this.pending = this.pending.slice(-this.MAX_QUEUE_SIZE)
      this.logger?.warn(`TickFence: queue overflow, dropped ${dropped} oldest interventions`)
    }
    return id
  }

  async beforeTick(): Promise<{ applied: number; remaining: number }> {
    if (this.inTick) throw new Error('TickFence: already in tick')

    const toApply = this.pending.slice(0, this.MAX_PER_TICK)
    this.pending = this.pending.slice(this.MAX_PER_TICK)
    this.inTick = true

    if (toApply.length > 0 && this.fenceCallback) {
      await this.fenceCallback(toApply)
    }
    return { applied: toApply.length, remaining: this.pending.length }
  }

  getQueueStatus() {
    return {
      pending: this.pending.length,
      max_per_tick: this.MAX_PER_TICK,
      estimated_clear_ticks: Math.ceil(this.pending.length / this.MAX_PER_TICK)
    }
  }
}
```

## 9. 风险控制：Plan B

### 9.1 触发条件

如果 Phase 3（LLM Orchestrator）发现 AutoGen 能力在 TypeScript 中难以复刻：

- 多轮 agent 对话的状态机复杂度超出预期
- AutoGen 特有的"反思-回环"逻辑难以维护

### 9.2 Plan B 方案：子进程内嵌推理微服务

```
主进程 (TypeScript)                           子进程 (Python)
┌──────────────────────────┐                 ┌──────────────────┐
│ SimulationEngine          │                 │ 推理微服务        │
│  ├─ tick dispatch ────────┼── stdin/JSON ──→│ 仅做 LLM 决策    │
│  ├─ needs decay           │                 │ 无状态、无路由    │
│  └─ event broadcast ←─────┼── stdout/JSON ──│ 单责任：推理     │
└──────────────────────────┘                 └──────────────────┘
```

### 9.3 Plan B 与 MCP 桥接的本质区别

| 维度 | MCP 桥接 | Plan B 子进程 |
|------|----------|--------------|
| 连接模型 | persistent connection | 按 tick 离散调用 |
| 失败影响 | 双系统耦合崩溃 | 居民仅"发呆"，主仿真继续 |
| 状态管理 | 共享状态 | 单行道数据流：状态快照 → 决策结果 |
| 协议开销 | HTTP/JSON-RPC 头开销 | stdin/JSON（≈0 开销） |
| 运维复杂度 | 双运行时监控 | 单运行时 + 进程守护 |

### 9.4 Plan B 实施要点

```typescript
// simulation/python-bridge.ts（仅 Plan B 启用时创建）

import { spawn } from 'child_process'

export class PythonReasoningBridge {
  private proc: ChildProcess

  constructor(private scriptPath: string) {
    this.proc = spawn('python', [scriptPath], {
      stdio: ['pipe', 'pipe', 'inherit']
    })
  }

  async decide(snapshot: ResidentSnapshot): Promise<ResidentAction> {
    return new Promise((resolve, reject) => {
      const timeout = setTimeout(() => reject(new Error('推理超时')), 30000)
      this.proc.stdout.once('data', (buf) => {
        clearTimeout(timeout)
        resolve(JSON.parse(buf.toString()))
      })
      this.proc.stdin.write(JSON.stringify(snapshot) + '\n')
    })
  }
}
```
